# SentiTrack 작업 요약 — 2026-07-21

## 1. 배경

이번 주(7/25 토요일 마감) SentiTrack 고도화 목표는 4가지였다.

1. 부정 리뷰가 긍정으로 표시되는 오표시 버그 수정
2. 배포 파이프라인 마이그레이션 재발 방지
3. KoELECTRA·Normalization·LLM 최종 평가(기존 실험 수치 검증 및 확정)
4. (다음 세션) README/포트폴리오용 수치 정리

오늘 하루 동안 위 1~3을 진행했고, 그 과정에서 실제 배포 장애 대응까지 이어졌다.

## 2. 운영 환경 재확인

작업을 시작하며 사용자가 준 EC2(3.39.241.113) 정보로 접속했으나, 해당 서버에는 SentiTrack 컨테이너가 전혀 없었고 `sentitrack.levelupseon.com`의 DNS도 다른 IP를 가리키고 있었다. 확인 결과 **실제 운영 서버는 OCI 인스턴스(168.110.109.236)**였다. EC2는 과거에 쓰이다 이관된 것으로 추정되며, 이후 모든 작업은 OCI 기준으로 진행했다.

## 3. 부정→긍정 오표시 버그 (커밋 `f4e0a09`)

- **제보**: "부정 리뷰가 긍정으로 표시되는 오류"
- **조사**: 운영 DB(OCI, 읽기 전용) 점검 결과 리뷰가 총 2건뿐이었고, 그중 id=2 `"제일 좋아하는 꽃향인데 조금 인공적인거 같아요ㅠㅠㅠ"`가 `POSITIVE 0.6702`로 저장되어 있었음.
- **원인 확정**: model_version 불일치(케이스 A)는 재현 결과 저장값과 완전히 일치해 배제. **케이스 B(모델 오분류)로 확정** — KoELECTRA 이진 분류기가 대조/혼합 문장을 문장 전체 단위로 판단하는 구조적 한계. 추가로 `experiments/clause_sentiment.py`의 `ENDING_CONNECTORS`에 **"인데"가 누락**되어 있어 해당 사례가 clause split조차 안 되고 있던 것도 함께 발견.
- **수정**: `ENDING_CONNECTORS`에 "인데" 추가, `main.py`의 `/predict`가 `analyze_clause_sentiment`를 통해 clause 단위로 재분류하고 `MIXED`를 반환하도록 연결. 재현 결과 id=2 사례가 `POSITIVE 0.6702 → MIXED 0.6702`로 정정됨.
- **부수 발견/수정**:
  - `python-inference/Dockerfile`이 `main.py`만 이미지에 복사하고 `experiments/`는 복사하지 않아, 신규 import가 배포 시 `ModuleNotFoundError`로 컨테이너를 죽였을 뻔한 것을 배포 전에 발견해 `COPY experiments ./experiments` 추가.
  - `frontend`의 `isPositive ? "POSITIVE" : "NEGATIVE"` 및 `app/me/page.tsx`의 `negativeCount = total - positiveCount` 패턴이 신규 `MIXED` 값을 전부 NEGATIVE로 오표시/오집계하는 구조라 같이 수정 (`types.ts`, `globals.css`, `ReviewCard.tsx`, `ArchiveCard.tsx`, `app/me/page.tsx`).
- **검증**: `python-inference` 테스트 **147개 통과**, gateway/frontend 빌드 성공, 운영 재현 테스트로 수정 확인.

## 4. 운영 DB 마이그레이션 상태 검증 (커밋 `f94254d`)

- `gateway/migrations/002_add_users.sql`, `003_add_scent_category.sql`이 `docker-entrypoint-initdb.d`로만 마운트되어 있어, 볼륨이 이미 존재하면 자동 반영되지 않는 구조적 문제를 점검.
- OCI 운영 DB 확인 결과 `sentitrack_reviews.user_id`, `sentitrack_products.scent_category` 컬럼은 **이미 반영되어 있었음** (볼륨이 002/003 추가 이후 재생성된 것으로 추정) — 별도 마이그레이션 실행은 불필요했음.
- 점검 과정에서 **운영 DB의 root 계정 비밀번호가 `.env`의 `DB_PASSWORD`와 불일치**하는 것을 발견 (`MYSQL_ROOT_PASSWORD`는 볼륨 최초 생성 시에만 적용되는데 이후 시크릿이 로테이션됨). 앱 계정(`sentitrack_user`)은 `sentitrack` 스키마에 대해 `ALL PRIVILEGES`를 보유하고 있어 마이그레이션 실행에 충분함을 확인.
- **재발 방지**: `deploy.yml`의 "Deploy to OCI via SSH" 스텝에 `gateway/migrations/*.sql`을 파일명 순서대로 순회하며 매 배포마다 재적용하는 반복문 추가, 계정은 root 대신 앱 계정(`secrets.DB_USER`/`secrets.DB_PASSWORD`)으로 우회 설정.

## 5. KoELECTRA·Normalization·LLM 최종 평가 (커밋 `6bdb4f4`, `8b5e869`)

### 5-1. 실제 `/predict` 기준 재측정에서 발견된 갭

기존에 기록된 오프라인 실험 수치(MIXED Recall 0.20 → 0.30, `clause_normalization.py`의 SIMPLE_DECLARATIVE 전략 적용 시)를, 오늘 수정된 실제 `/predict` 엔드포인트(40건 데이터셋, `evaluate_predict_endpoint.py`)로 재측정한 결과 **MIXED Recall이 0.20**으로 나왔다. 원인은 `/predict`가 clause split(`clause_sentiment.py`)만 연결됐고, recall을 0.30으로 올렸던 정규화 단계(`clause_normalization.py`)는 배선되지 않았기 때문이었음.

### 5-2. clause_normalization을 `/predict`에 마저 연결

- `analyze_clause_sentiment()`에 `clause_normalizer` 선택적 인자 추가(기본값 `None`이면 기존 동작과 동일), 절 예측 직전에만 적용, baseline 전체 문장 예측에는 미적용.
- `main.py`에서 `SIMPLE_DECLARATIVE` 전략을 연결하는 wrapper 추가.
- 신규 테스트 3개 추가, 전체 **150개 테스트 통과**(147+3).
- 재측정 결과: **MIXED Recall 0.20 → 0.30**, POSITIVE/NEGATIVE 정확도 **1.0 유지(회귀 없음)**, **false MIXED 0건**(신규 오탐 없음).

### 5-3. LLM v1+taxonomy 재확인 중 캐시 재현성 버그 발견

- 대표 12건을 캐시(`llm_sentiment_cache.jsonl`, `--use-cache`, 실제 API 호출 0건)로 재실행한 결과 overall 정확도가 11/12(0.9167)로, 기존 기록(1.0)과 차이가 남.
- 원인 조사 결과, `JsonlLLMCache`가 **같은 cache_key에 대해 나중에 쓴 값이 이전 값을 덮어쓰는(last-write-wins)** 구조였고, 세션 간 재호출된 비결정적 LLM 응답이 원래 캐시를 덮어쓴 것으로 확인. **taxonomy 정규화 코드 자체의 결함이 아니라 캐시 재사용의 재현성 한계**로 결론.
- 기존 기록(정확도 1.0, taxonomy 정규화 aspect F1 `0.9032`)은 단일 세션 실측치로서 유효하게 유지.

### 5-4. 40건 전체 평가 — API 키 부재로 부분 검증

- 이 환경에 `SENTITRACK_LLM_API_KEY`/`MODEL`/`BASE_URL`이 프로세스/사용자/머신 모든 범위에서 미설정 확인 — 실제 API 호출이 원천적으로 불가능한 상태.
- 가짜 자격증명으로 OpenRouter에 무의미한 요청을 보내는 대신, 네트워크 호출을 완전히 차단하는 신규 스크립트(`evaluate_llm_cache_only.py`)로 캐시에 있는 항목만 안전하게 검증.
- 결과: 40건 중 **31건만 캐시 존재, 실제 평가**(9건은 시도조차 안 함) — **MIXED Precision/Recall/F1 = 1.0/1.0/1.0** (10/10 캐시 존재, 전부 정답). Overall 정확도 0.9355(29/31). Aspect name F1(raw/normalized) 0.095/0.615(캐시 오염 영향, 5-3 참고).

## 6. 배포 동시 경합 장애 대응 (실전 사건, 커밋 `f54c433`)

- **증상**: `6bdb4f4`, `8b5e869`가 6분 간격으로 연달아 push되면서 `deploy.yml`에 동시 실행 제어가 없어 **두 워크플로가 같은 OCI 서버에 SSH 배포를 동시에 실행**. 서버 load average가 **9.56**까지 상승.
- **원인**: 두 프로세스가 같은 Docker 이미지를 동시에 pull하며 자원 경합. 오래된 워크플로(`6bdb4f4`)를 GitHub Actions에서 취소하자, 진행 중이던 이미지 레이어 추출 작업이 중간에 끊기면서 containerd에 **dangling lease**(정리되지 않은 임시 리소스)가 남았고, 남아있던 다른 pull(`8b5e869`)이 같은 레이어를 기다리며 CPU 사용률 0%로 무응답 상태가 됨.
- **대응**: `8b5e869` 워크플로도 정상 취소를 시도했으나 GitHub 쪽에서는 이미 타임아웃 없이 자연 완료(success) 처리된 상태였고, 실제로 서버에서 확인한 결과 pull이 그 사이 스스로 풀려 배포가 정상 완료되어 있었음 (컨테이너 재생성, git HEAD `8b5e869` 확인, 마이그레이션 스텝도 에러 없이 통과, 로컬에서 검증한 신규 MIXED 사례가 운영에서도 동일하게 재현됨).
- **재발 방지**: `.github/workflows/deploy.yml`에 `concurrency: { group: deploy-production, cancel-in-progress: false }` 추가. `cancel-in-progress: false`로 설정한 이유는, 먼저 실행 중인 배포를 강제 취소하면 오늘처럼 dangling 리소스가 남을 수 있어, 대신 뒤에 트리거된 워크플로를 대기열에 넣어 **순차 실행**되도록 하기 위함.
- **검증**: YAML 문법 파싱 확인 후 반영. 이 커밋 자체가 배포를 트리거했고, 경쟁 없이 단독 실행되어 **success**로 완료. 배포 후 서버 load average는 **0.17**로 정상화됨.
